# Secrets management — rotation procedures

The app reads its secrets from **AWS Systems Manager Parameter Store**, NOT from a `.env` file checked into git. The EC2 instance role granted by Phase 1 Terraform allows `ssm:GetParametersByPath` on `/karha/prod/*`.

## What's where

| Parameter | Type | Used by | How rotated |
|-----------|------|---------|-------------|
| `/karha/prod/JWT_SECRET` | SecureString | App (signs JWTs) | Quarterly. See below. |
| `/karha/prod/CRON_SECRET` | SecureString | App (gates cron endpoint) | Quarterly with JWT_SECRET. |
| `/karha/prod/DATABASE_URL` | SecureString | App (Phase 4+) | When the DB password rotates. |
| `/karha/prod/DB_MASTER_PASSWORD` | SecureString | Ops only | RDS master rotation flow. |
| `/karha/prod/STORAGE_BACKEND` | String | App (Phase 3+) | Manual; values: `local` or `s3`. |
| `/karha/prod/S3_BUCKET` | String | App (Phase 3+) | Manual; bucket name. |
| `/karha/prod/UPLOADS_PUBLIC_URL` | String | App (Phase 3+) | Optional CloudFront domain override. |

Razorpay + Google OAuth keys are stored differently — they live in the app's `site_config` table and are managed from `/admin → Integrations`. That decision was made deliberately: those credentials are rotated *frequently* by the brand team (when a Razorpay key gets compromised, you go to the Razorpay dashboard and reissue), and Parameter Store changes need shell access. Keeping them in the DB lets a non-engineer rotate them.

## Rotate JWT_SECRET

Customer JWTs are valid for 7 days; admin JWTs for 24 hours. Rotating the secret invalidates *every* outstanding token, forcing all logged-in users to log in again. Fine for low-traffic launch; mildly annoying once you have thousands of customers.

Procedure:

1. Generate a new value: `openssl rand -hex 32`
2. Update Parameter Store:

   ```bash
   aws ssm put-parameter --name /karha/prod/JWT_SECRET \
     --value "<new-hex-value>" --type SecureString --overwrite
   ```

3. Reload PM2 to pick up the new value (the app reads at boot via `scripts/load-secrets.sh`):

   ```bash
   ssh ec2-user@karha.in
   sudo -u karha bash -c "
     cd /home/karha/app
     bash /home/karha/karha-infra/scripts/load-secrets.sh
     pm2 reload karha --update-env
   "
   ```

4. Verify a fresh login works at `/admin`. Existing sessions get a "session expired" prompt on their next request — expected.

For a *zero-downtime* rotation (no forced re-login), see "Staggered rotation" below.

## Staggered rotation (no forced re-login)

To rotate JWT_SECRET without invalidating outstanding tokens:

1. Add a second Parameter Store entry: `/karha/prod/JWT_SECRET_NEXT`
2. Deploy a release of `karha-web` that verifies tokens against *either* `JWT_SECRET` or `JWT_SECRET_NEXT` (signs only with `JWT_SECRET`). This is a small change in `src/lib/auth.ts` — verify against both, sign against the primary.
3. Wait 7 days for the longest-lived (customer) tokens to roll over.
4. Promote: rename `JWT_SECRET_NEXT` → `JWT_SECRET` in Parameter Store. Reload PM2.
5. Deploy a release that drops the dual-verify code (returns to single-secret).

This is overkill for launch. Implement the staggered flow only when forced re-login becomes painful.

## Rotate the RDS master password (Phase 4+)

RDS supports automatic password rotation via Secrets Manager. Until that's wired:

1. Console → RDS → karha-prod → Modify → set new master password → Apply immediately.
2. Update Parameter Store: `aws ssm put-parameter --name /karha/prod/DB_MASTER_PASSWORD --value "<new>" --type SecureString --overwrite`
3. The `karha_app` role's password is independent and unaffected — only the `karha_admin` master password changes.

## Rotate the karha_app role password (Phase 4+)

This one *does* affect the running app:

1. From the EC2 box, log into Postgres as karha_admin and `ALTER USER karha_app WITH PASSWORD '<new>';`
2. Update `/karha/prod/DATABASE_URL` with the new password.
3. `pm2 reload karha --update-env` — momentary connection blip while the app reconnects.

## Rotate Razorpay or Google OAuth credentials

In-band, no shell access needed:

1. Sign in to `/admin → Integrations`
2. Paste the new key in the appropriate field
3. Click Save

Server picks up the change on the next request.

## Storage of secrets — what NOT to do

- Don't put secrets in `.env.local` checked into git. (`.env.production` lives only on the EC2 box, generated by `scripts/load-secrets.sh` from Parameter Store.)
- Don't put secrets in Terraform state files. The `random_id` resources have `ignore_changes` lifecycle blocks so a routine `terraform apply` never rotates them. State files (S3-backed in a future improvement) are encrypted but should never carry production secrets.
- Don't grant `ssm:*` to the EC2 role — Phase 1 narrows it to `ssm:Get*` on `/karha/prod/*` so a compromised box can't read other accounts' secrets.
- Don't share the `.pem` file. If it leaks, rotate the EC2 key pair (`aws ec2 create-key-pair --key-name karha-prod-2 …` + console swap).
